# 软件功能报价方法论

> answer 工作流在「软件报价/功能估价」场景下的领域适配参考。
> 从《兴义水务功能报价清单》实战中提炼，可复用于类似任务。

---

## 适用场景

- 基于 PRD/需求文档对软件功能点进行逐项定价
- 客户面需呈现透明报价清单，内部需保留评估依据
- 预算池已知（如 ¥50k 服务 + ¥180k 开发），需合理分摊到功能点

---

## 方法论组合

### 复杂度评估：Fibonacci 点数（1/2/3/5/8）

| 点数 | 标签 | 判定标准 |
|------|------|---------|
| 1 | 极简 | 单一表单/页面，无复杂逻辑，纯 CRUD |
| 2 | 简单 | 表单+基础校验+简单审批流，1-2 表关联 |
| 3 | 中等 | 多步骤流程、条件分支、多表联动、报表聚合 |
| 5 | 复杂 | 核心算法引擎、多实体协调、复杂状态机、集成对接 |
| 8 | 极重 | 系统级基础设施、跨模块贯穿、高技术风险 |

### 价值评估：RICE 模型（R × I × C / E）

| 因子 | 含义 | 1-5 分锚点 |
|------|------|-----------|
| Reach | 影响人数 | 1=单人 → 5=全公司+外部 |
| Impact | 影响强度 | 1=微改善 → 5=变革性 |
| Confidence | 信心 | 1=低（需求模糊） → 5=极高（可复用） |
| Effort | 实施成本 | **直接取 Fibonacci 点数** |

### 定价公式

```
① Fib基础单价 = 预算池 ÷ Σ该池Fib点数
② 单项基础价 = Fib基础单价 × 该功能Fib点数
③ RICE浮动：按RICE得分在同池内的分位，映射到浮动比例：

   RICE分位       浮动比例
   ─────────     ──────────
   前20%（高分）   +27.5%
   20-40%         +17.5%
   40-60%          0%
   60-80%         -17.5%
   后20%（低分）   -27.5%

④ 单项原始价 = 基础价 × (1 + 浮动比例)
⑤ 池内归一化：Σ原始价缩放至预算池精确值（允许 ±1% 舍入，再手动补零头）
```

## Excel 产出标准

### Sheet1 — 客户面（报价清单）

| 列 | 字段 | 说明 |
|----|------|------|
| A | 序号 | 1-N |
| B | 功能域 | 按域分组，域间用分隔行 |
| C | 功能点名称 | 第三级模块页面粒度 |
| D | 功能简述 | 一句话（≤30字） |
| E | 价格（¥） | 整数，千分位格式 |

- 冻结表头
- 域分隔行带浅色底色
- 底部合计行
- **不出现** Fibonacci/RICE/浮动比例等内部数据

### Sheet2 — 内部（完整评估表）

| 列 | 字段 | 说明 |
|----|------|------|
| A | 序号 | |
| B | 功能域 | |
| C | 功能点名称 | |
| D | Fibonacci | 1/2/3/5/8 |
| E | Reach | 1-5 |
| F | Impact | 1-5 |
| G | Confidence | 1-5 |
| H | RICE得分 | =E×F×G/D，一位小数 |
| I | 基础价 | Fib比例分摊价 |
| J | 浮动比例 | +27.5% ~ -27.5% |
| K | 最终价 | |
| L | 预算池 | 服务/开发 标注 |
| M | 备注 | 特殊调整原因 |

- RICE ≥25 绿色底色，≤10 红色底色
- 浮动正数绿色字体，负数红色字体
- 顶部标题注明方法论和浮动规则
- 底部附评分锚点说明

## 实施流程（嵌入 answer 7 阶段）

1. **Phase 1 Clarify**：确认粒度（几级模块）、预算池、方法论、Sheet 结构
2. **Phase 2 Brief**：锁定定价公式、浮动规则、Excel 列定义
3. **Phase 3 Architect**：从 PRD 提取全部功能点，建三级结构树
4. **Phase 4 Standards**：定义 Fib/RICE 评分锚点 + 浮动映射规则
5. **Phase 5 Decompose**：Task 1 评分 → Task 2 计算+预览 → Task 3 生成 Excel
6. **Phase 6 Build**：用 `execute_code`（Python + openpyxl）生成 .xlsx
7. **Phase 7 Review**：对照 Brief 逐项检查完整性/一致性/标准合规

## 常见陷阱

| 陷阱 | 避免方式 |
|------|---------|
| 总价与预算池偏差大 | 加归一化步骤（缩放因子），最终手动补零头到精确值 |
| 域内 RICE 区分度不足 | 服务成果类文档适当拉大 Impact 分差（核心制度 vs 辅助文档） |
| 客户面出现内部数据 | 严格分 Sheet，Sheet1 只保留 5 列 |
| Fibonacci 和 RICE 互相污染 | 先打 RICE 四因子，再打 Fib（Effort 直接复用），避免预期影响 |

## 预算池扣除模式（总预算 ≠ 开发预算）

当客户总预算包含非开发费用（订阅、Token、License、硬件等）时，**必须先扣除刚性支出，剩余部分才参与 Fib 分摊**。

### 识别与扣除步骤

1. 从需求文档/合同中提取所有非开发费用项
2. 对每项给出低/中/高三种估算，取中值（或用户确认值）
3. `开发预算 = 总预算 − Σ刚性支出`
4. 开发预算参与 Fib 分摊；刚性支出在 Excel Sheet1 底部单独列出

### 示例：浙能燃气（¥400,000）

```
总预算 ¥400,000
  ├── 钉钉企业版订阅   ¥30,000–50,000
  ├── AI Token 调用费  ¥50,000–100,000
  ├── 刚性合计（中值）  ¥120,000
  └── 开发预算         ¥280,000（参与 Fib 分摊）
```

### 常见刚性支出类型

| 类型 | 示例 |
|------|------|
| 平台订阅 | 钉钉/企微/飞书企业版、云服务 |
| Token/API 调用 | 大模型推理 Token、第三方 API 按量付费 |
| License | OCR 引擎、BI 工具、中间件 |
| 硬件 | GPU 服务器（如未自备） |

## 平台替代降本模式

当客户已有成熟平台（钉钉/企业微信/飞书等），可用平台原生能力替代自研模块，**降低对应功能点的 Fibonacci 复杂度**。

### 替代映射速查

| 平台能力 | 可替代的自研需求 | Fib 降幅 |
|---------|-----------------|:-------:|
| 知识库（钉钉/飞书） | 自研文档存储+权限分离+全文检索 | 平台基础 -1 |
| SSO 单点登录 | 自研统一身份认证集成 | 平台基础 -1 |
| 组织架构同步 | 自研 LDAP/AD 同步服务 | 平台基础 -1 |
| AI Agent 平台（钉钉悟空等） | 自研 Agent 编排+工具注册+MCP 网关 | 平台基础 -2 |
| MCP/API 网关（平台内置） | 逐个系统对接开发（10+ 内网系统网页查询） | 数据治理/办公助手等 -1~2 |

### 操作流程

1. **Phase 1 Clarify** 阶段确认客户平台生态（钉钉/企微/飞书/自研）
2. 逐一评估：平台原生能力能否替代需求？非保密信息优先用平台
3. 调整受影响功能点的 Fibonacci 分（降 1-2 点）
4. 降本结余优先用于将中度项提升为深度

### 重要约束

- 平台 API 封装由企业方提供时，不计入预算（如 MCP/API 接口企业自行开发）
- 仅替代不影响功能交付质量的部分；核心业务逻辑、AI 推理、知识检索仍由 Agent 侧负责
- 平台订阅费在预算池扣除步骤中已处理（见上方「预算池扣除模式」）

## 平台工作坊交付模式（≠ 传统开发交付）

当客户预算不足以覆盖全部功能点的传统开发时，且存在成熟的低代码/AI Agent 平台（如钉钉 DEAP、扣子、Dify），可以采用**平台订阅 + 工作坊培训**替代**全自研开发**。

### 模式对比

| 维度 | 传统开发交付 | 平台工作坊交付 |
|------|------------|-------------|
| 预算构成 | 开发人力费占大头 | 平台订阅(年费) + Token + 工作坊(培训) |
| Agent 形态 | 乙方开发交付成品 | 客户团队在平台上自建 |
| 覆盖面 | 预算限制定量（RICE 排序取舍） | 全量覆盖（客户后续自建剩余Agent） |
| 知识转移 | 浅度（操作培训） | 深度（客户掌握工具和搭建能力） |
| 长期维护 | 乙方维保（额外年费） | 客户自维护 |
| 工期 | 16周+ | 2-3天集中工作坊 + 后续自建 |

### 预算公式

```
总预算 = 平台订阅年费 + Token估算 + 工作坊费用
        （平台费 = 基础平台 + 必需付费应用/插件）

工作坊费用 ≈ ¥80,000–150,000（2-3天，含前期需求梳理+现场培训+产出验收）
```

### 工作坊设计模板

| 阶段 | 内容 | 产出 |
|------|------|------|
| **前期**（远程，1周） | 需求梳理、知识库文档收集、DEAP 环境预配置、MCP 插件预对接 | 知识库清单 + Agent 优先级列表 |
| **Day 1** | DEAP 平台认知 + 知识库搭建（16个模块分批）+ 第一个 Agent 实战 | 2-3个知识库上线 + 1个示范 Agent |
| **Day 2** | Agent 开发实战（核心 6-8 个 Agent 分组搭建）+ MCP 插件配置 | 6-8个核心 Agent 完成 |
| **Day 3** | 进阶技能 + 系统对接实战 + 剩余 Agent 框架搭建 + 产出验收 | 全部 Agent 框架 + 验收清单 |
| **后期**（远程，2周） | 问题答疑 + Agent 优化 + 交付文档 | 最终验收报告 |

### 适用判断条件

满足以下**任意 2 条**，优先考虑平台工作坊模式：

1. Agent 数量 ≥ 10 个，传统开发无法全覆盖
2. 客户 IT 团队有能力学习和维护低代码平台
3. 存在成熟的生态平台（钉钉 DEAP/扣子/Dify）且平台费 ≤ 总预算 60%
4. 客户有长期自主扩展 Agent 的意愿
5. 需求中包含大量 RAG 知识库 + 系统对接（平台内置可大幅降本）

> 📚 **DEAP 平台详情**见 `references/dingtalk-deap.md` — 钉钉企业AI平台完整模块矩阵和费用参考。

### 与 RICE 的衔接

- 平台工作坊模式下，**RICE 评分用于确定工作坊中的 Agent 搭建优先级**（Day2 先搭 RICE 高分项）
- 传统开发模式下，**RICE 评分用于确定预算分配深浅**
- 两种模式应在 Phase 1 Clarify 阶段同时评估，形成双方案对比

## 评估报告结构规范

用户偏好：评估报告的**业务内容前置，方法论后置**。结构顺序：

```
1. 需求-功能匹配清单     ← 客户最关心的：哪些需求对应哪些功能
2. 场景功能实现路径       ← 怎么做
3. 功能边界描述          ← 做什么/不做什么
4. 方案对比（如有多个方案） ← 决策依据
5. [工作坊设计]（如适用）  ← 交付内容
6. 预算结构              ← 钱怎么花
7. 评估方法论（Fib/RICE） ← 评分依据（后置）
8. 交付物 + 后续行动
```

理由：客户和决策者先看「能做什么、怎么做」，再看「怎么评估的」。方法论是乙方内部工具，不应占据报告前几页。
