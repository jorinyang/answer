# PRD 撰写模式（Answer 工作流适配）

> 从《万峰林景区基建环卫综合管理系统》PRD 撰写中提炼，适用于需要综合会议纪要、录音转写、客户反馈、竞对信息等多来源材料，输出可直接用于开发的完整PRD的场景。

---

## 触发信号

- 用户明确说「用 answer 写 PRD」
- 用户提供了会议纪要 + 录音转写 + 流程图等多来源材料
- 需要从零整理一份「开发人员拿到就能开工」级别的完整需求文档

---

## 前置阶段：材料收集

在进入 Phase 1 Clarify 之前，必须先完成材料收集：

1. **逐份确认收到** — 用户提供的每份材料（纪要、转写、流程图、竞对截图）都要明确告知「已解析」
2. **竞对截图逐张记录** — 用户可能逐张发送竞对方案截图。每收到一张先详细解析，记录功能点/价格/平台/差异化，等用户发下一张。不催促，不跳过
3. **材料交叉验证** — 对比纪要摘要和录音转写全文，找出纪要中简化的细节和遗漏的讨论点

---

## Phase 1 Clarify 适配：缺口分析模式

当 PRD 已有初稿（如 v1.0）且用户提供了新的参考材料时，Clarify 阶段不是从零追问，而是：

1. **对照已有一稿 + 新材料**，逐项列出：
   - ✅ 已覆盖的内容
   - ❌ 新发现但未覆盖的内容
2. **按严重程度分级**：
   - **P0** — 不补影响方案可行性（业务流程逻辑缺失、预算约束不明确）
   - **P1** — 不补影响方案竞争力（服务条款、付款方式、差异化对比）
   - **P2** — 锦上添花（细节优化、非核心功能说明）
3. **追加确认边界问题** — 对新增模块中需要用户决策的点（如巡检是否需要自动计划？二维码是否需要打印？报修派单逻辑？），逐条列出等用户确认
4. **输出缺口清单 + 确认问题**，等用户回复后再进入 Build

## PRD 标准章节结构

```
第一章   项目概述（背景/目标/范围/方案定位）
第二章   角色与权限（定义 + 权限矩阵 + 登录机制）
第三章   业务流程（核心流程 + 状态机 + 异常分支 + 新增子流程）
第四章   功能需求（按模块逐功能点，含双路径覆盖标记）
第五章   数据模型（ER图 + 核心表DDL）
第六章   技术方案与实施（双路径对比 + 周期 + 交付物 + 付款 + 售后）
第七章   非功能性需求（性能/安全/兼容性/二维码规格）
附录     术语表/设备清单/竞对差异化
```

---

## 关键原则

### 功能表中必含"双路径覆盖"列

| 编号 | 功能点 | 描述 | 角色 | 轻量方案 | 完整方案 |
|:---:|--------|------|------|:--:|:--:|

让客户一眼看到自己在每个价位得到什么。

### 业务流程中必含异常分支

不只是画主流程。必须包含：
- 库存不足时谁触发采购？（库管→采购申请，非工人）
- 审批驳回后怎么走？（退回员工调整→重新提交）
- 验收不达标？（退回重做+原因）

### 预算必须与Excel/Word同步

PRD中的价格数字（如果包含）必须与 Excel 报价清单 + Word 技术方案中的预算表完全一致。三份文档的价格是同一套数据的不同呈现。

### "先轻后重"策略显式化

如果方案支持渐进式建设，必须在 PRD 中明确：
- 阶段一的交付范围和价格
- 阶段二的触发条件
- 迁移成本（如 1/5 原开发费）

---

## 与竞对方案对比的写法

- 用事实，不用贬低词汇
- 对比维度：平台生态（钉钉 vs 微信小程序）、数据所有权、迭代方式、售后服务模式
- 竞对价格仅作锚点记录，不在 PRD 中直接对比"我们便宜XX%"
- 附录中列出差异化，正文中不集中讨论竞对
