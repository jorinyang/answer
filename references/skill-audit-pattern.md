# Skill Audit Pattern — 用 answer 校验已有技能

当用户需要对一个**已存在的技能/方案/系统**做结构化审计和重新设计时，按此模式执行。

## 适用场景

- 用户说「用 answer 重新校验 XX 技能」
- 用户说「帮我审查这个方案是否完整」
- 已有的技能/方案缺少设计文档，需要补全 Brief/Architecture

## 与标准 7 阶段的区别

标准 answer 是从零构建。审计已有技能时，**不需要重跑完整 7 阶段**。技能的技术骨架已经存在，缺少的通常是 Clarify 决策记录和 Brief 设计叙事。

## 审计流程（4 步）

### Step 1: 完整读取已有产物

用 skill_view 读取 SKILL.md + 所有 scripts + references。了解当前实现的完整状态。

### Step 2: Clarify 逆向工程

遍历已有代码和文档，提取所有**隐式设计决策**——代码做了假设但 SKILL.md 从未回答的问题。

组织为三层决策树：

| Tier | 性质 | 例子 |
|------|------|------|
| 🔴 Tier 1 | 不改会导致返工 | 客户什么时候打开页面？离线能力？ |
| 🟡 Tier 2 | 改了显著提升体验 | 个性化程度？分享策略？ |
| 🟢 Tier 3 | 长期架构考虑 | 生命周期管理？多页面？ |

每个问题标注**"为什么重要"**和**"当前隐含假设"**，让用户理解不回答的后果。

### Step 3: 收集答案 → 收敛设计

用户回答 Tier 1 问题后，设计就会收敛一个方向。可能有剩余开放问题，用选项式追问（A1/A2、B1/B2）让用户快速选择。

### Step 4: 实施更新

按收敛后的设计更新所有技能文件。优先顺序：
1. schema/references（数据模型）
2. 提取脚本（数据来源）
3. 生成脚本（产出逻辑）
4. SKILL.md（文档最后更新，反映所有变更）

## 关键技巧

- **实体验证优先（🔴 核心原则）**：不信任文档中的声明。对每个声称"✅ 已实现"的能力，运行真实的工具调用来验证。技能文档写 `✅ CRM 查询` 不代表查询命令能跑通——必须 `--dry-run` 或实际调一次。文档中的 token、command、API endpoint 全部做存活验证。
- **选项式追问**：对开放问题提供 2-4 个具体选项，标注各自影响。用户只需回复 A2/B1/C2，极简决策。
- **"为什么重要"列**：每个 Clarify 问题必须解释不回答的后果，否则用户看不到价值。
- **"当前隐含假设"列**：指出代码已经做了什么假设，让用户知道不改 = 默认走这个方向。
- **验证清单**：
  1. 硬编码的 token/ID → `lark-cli` 或 curl 查存活
  2. MCP 工具可用性 → 对照 `config.yaml` 中的 MCP 配置验证技能文档声称的工具是否存在
  3. 命令语法 → `--dry-run` 或最小参数调用
  4. 格式兼容性 → 模板格式 vs 实际渲染工具的能力边界
