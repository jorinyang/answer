# Domain Mapping: 领域到 7 阶段的参数化模板

> 当 answer 识别到不同领域时，加载对应的追问清单和模板变体。

---

## 领域识别规则

| 信号词 | 识别为 |
|--------|--------|
| 商业模式、产品、市场、营收、获客、增长、用户、启动、功能、模块 | 🏢 新业务/产品 |
| 技能、skill、工作流、Hermes、工具、CLI | 🛠️ 新技能 |
| 方案、计划、提案、策略、规划、PPT、汇报、BP、建议书、立项 | 📋 新方案 |
| 流程、SOP、标准化、规范、制度、操作手册、优化 | 🔄 流程/SOP |
| 分析、调研、研究、竞品、拆解、梳理、可行性 | 🔍 分析/调研 |
| 决策、对比、选择、权衡、几个方案、怎么选 | 🎯 决策支持 |

## 🏢 新业务构建

### Phase 1: Clarify — 追问清单

```markdown
1. 核心用户是谁？他们现在的替代方案是什么？
2. 用户为什么需要这个？（不是"用户说需要什么"，是 JTBD）
3. 什么算成功？3 个月/12 个月的具体指标？
4. 最不确定的假设是什么？（需求真伪？付费意愿？执行可行性？）
5. 团队有什么独特优势？对手不能复制的是什么？
6. 预算/时间/团队约束？
7. 明确不做什么？（容易分散注意力的相邻市场）
```

```markdown
## 商业模式
- 收入来源：
- 成本结构：
- 关键指标：

## 风险评估
- 最大风险：
- 缓解策略：
```

### Phase 3: Architect — 追加模块

```markdown
## 业务流程总览
{流程图或步骤序列}

## 组织角色
| 角色 | 职责 | 所需能力 |
|------|------|---------|
```

### Phase 4: Standards — 追加规范

```markdown
## 业务规则
| 规则 | 适用场景 | 例外情况 |
|------|---------|---------|
```

### Phase 5: Decompose — 里程碑导向

```markdown
## Milestone 1: {M1 名称}（Week 1-2）
- [ ] 任务 1
- [ ] 任务 2

## Milestone 2: {M2 名称}（Week 3-4）
...
```

## 🛠️ 新技能创建

### Phase 1: Clarify — 追问清单

```markdown
1. 谁会在什么场景下触发这个技能？
2. 输入是什么？输出是什么？
3. 依赖哪些已有技能/工具/API？
4. 能力边界——明确不覆盖哪些场景？
5. 和已有技能的关系？（增强/替代/互补）
6. 最常见的错误使用方式是什么？
7. 有没有可以参考的已有技能？
```

### Phase 2: Brief — 追加字段

```markdown
## 技能接口
- 触发条件：
- 输入参数：
- 输出格式：
- 错误处理：

## 依赖关系
| 技能/工具 | 关系 | 备注 |
|----------|------|------|
```

### Phase 3: Architect — 追加模块

```markdown
## 执行流程（工作流）
{步骤序列，含决策分支}

## 子技能/子函数
| 子模块 | 职责 | 输入/输出 |
|--------|------|----------|
```

### Phase 4: Standards — 追加规范

```markdown
## 命名约定
- 技能名：kebab-case，≤64 字符
- 函数名：snake_case
- 文件路径：absolute

## 代码风格
- Python：black formatter, 88 字符
- 错误消息：中文，具体到参数

## 技能文档结构
- YAML frontmatter (name, description, triggers, version)
- 功能概述
- 执行流程
- 限制条件
- 常见错误
- 引用文件清单
```

### Phase 5: Decompose — 技能导向

```markdown
## Foundation
- [ ] 创建技能骨架（SKILL.md frontmatter + 目录结构）
- [ ] 定义 reference 文件和模板

## Core
- [ ] 实现核心逻辑（按 Architect 的流程图）
- [ ] 错误处理 + 边界情况

## Polish
- [ ] 端到端测试（真实场景）
- [ ] Review: 对照 Brief 检查
```

### Phase 6: Build — 调用技能

- `skill_manage(action='create')` — 创建技能
- `skill_manage(action='write_file')` — 添加引用文件
- 终端直接测试触发

### Phase 7: Review — 追加维度

```markdown
## 技能验证
| 测试场景 | 预期结果 | 实际结果 | 通过 |
|---------|---------|---------|------|
```

## 📋 新方案规划

### Phase 1: Clarify — 追问清单

```markdown
1. 这份方案的受众是谁？他们用什么标准判断方案好坏？
2. 方案通过的关键决策人是谁？他/她最关心什么？
3. 截止日期？产出形式？（飞书文档/PPT/HTML/混合）
4. 最容易被 challenge 的点是什么？
5. 需要引用的数据/案例在哪里可以找到？
6. 明确不涉及的范围？
7. 有没有可以参考的已有方案？
```

### Phase 2: Brief — 追加字段

```markdown
## 受众分析
| 受众 | 角色 | 关注点 | 理解水平 |
|------|------|--------|---------|
```

### Phase 3: Architect — 追加模块

```markdown
## 方案逻辑链
{核心论点 → 支撑论据 → 数据/案例 → 实施路径 → 预期效果}

## 方案结构（目录）
1. 背景/问题
2. 方案概述
3. 详细方案
   3.1 模块A
   3.2 模块B
4. 实施计划
5. 风险与对策
6. 预算/资源需求
7. 预期效果/KPI
```

### Phase 4: Standards — 追加规范

```markdown
## 文档格式规范
- 标题层级：≤ 3 级
- 关键数据必须引用来源
- 每个论点 ≤ 3 个支撑证据

## 内容规范
- 数字突出（大字号/加粗）
- 复杂概念用可视化（SVG/表格/流程图）
- 方案 vs 执行的界限清晰
```

### Phase 5: Decompose — 章节导向

```markdown
## Foundation
- [ ] 收集所需数据/案例
- [ ] 搭建文档骨架（按 Architect 的结构）

## Core
- [ ] 写背景/问题章节
- [ ] 写详细方案（每个模块一个任务）
- [ ] 写实施计划
- [ ] 写预算/KPI

## Polish
- [ ] 蓝军审查（调 blue-team）
- [ ] 可视化增强（图表/架构图）
- [ ] 格式统一 + 终审
```

## 🔄 流程/SOP 设计

### Phase 1: Clarify — 追问清单

```markdown
1. 这个流程的核心目标是什么？（效率？合规？体验？）
2. 谁执行这个流程？他们的当前痛点是什么？
3. 流程的起点和终点是什么？
4. 有多少个步骤？哪些是必须的？哪些是可选的？
5. 哪些步骤容易出错或绕过？
6. 有没有例外情况？如何处理？
7. 已有的相关流程/规范是什么？
```

### Phase 2: Brief — 追加字段

```markdown
## 流程目标
- 核心目标：
- 当前痛点：
- 优化后预期：

## 适用范围
- 适用角色：
- 适用场景：
- 不适用的例外：
```

### Phase 3: Architect — 追加模块

```markdown
## 流程图（步骤序列）
1. 步骤 A → 判断条件 → A1/A2
2. 步骤 B
...

## 角色与职责
| 角色 | 步骤 | 权限 | 工具/表单 |
|------|------|------|----------|

## 异常路径
| 异常场景 | 处理方式 | 升级路径 |
|---------|---------|---------|
```

### Phase 4: Standards — 追加规范

```markdown
## 命名规范
- 步骤编号：数字序列，异常步骤用字母后缀（3a, 3b）
- 角色命名：职位名称，不用人名

## 格式规范
- 流程图：ASCII art 或 mermaid
- 表单字段：必填标注 *
- SLA 用具体时间（≤2小时），不用模糊词（"尽快"）
```

### Phase 5: Decompose — 步骤导向

```markdown
## Foundation
- [ ] 梳理现有流程（如有）
- [ ] 访谈执行角色

## Core
- [ ] 画流程图（正常路径）
- [ ] 定义异常路径
- [ ] 设计表单/模板

## Polish
- [ ] 内部走查（模拟执行）
- [ ] Review: 对照 Brief 检查
```

### Phase 6: Build — 调用技能

- `feishu-doc` — 在飞书创建结构化 SOP 文档
- `feishu-table` — 如需流程跟踪表

```markdown
|------|------------|------|------|

## 易用性检查
| 检查项 | 结果 |
|--------|------|
| 新手能否独立完成？ | |
| 异常路径是否覆盖？ | |
```

## 🔍 分析/调研

### Phase 1: Clarify — 追问清单

```markdown
1. 分析/调研的目的是什么？（决策支持？知识积累？趋势判断？）
2. 分析结果的受众是谁？他们最关心什么结论？
3. 数据/信息从哪里获取？可获取性？
4. 分析框架是什么？（SWOT/PEST/波特五力/自定义）
5. 时间范围？分析的深度要求？
6. 关键的对比维度是什么？
7. 最可能无法获取的信息是什么？如何应对？
```

### Phase 2: Brief — 追加字段

```markdown
## 分析目标
- 目的：
- 受众：
- 决策用途：

## 数据源
| 来源 | 类型 | 可获取性 | 可信度 |
|------|------|---------|--------|

## 关键假设
| 假设 | 验证方法 |
|------|---------|
```

### Phase 3: Architect — 追加模块

```markdown
## 分析框架
{选用的分析框架及维度说明}

## 数据收集计划
| 数据项 | 来源 | 收集方法 | 负责人 | 截止 |
|--------|------|---------|--------|------|

## 报告结构
1. 执行摘要（结论先行）
2. 背景与目的
3. 方法论
4. 分析发现（按维度）
5. 关键洞察
6. 建议/下一步
```

### Phase 4: Standards — 追加规范

```markdown
## 数据引用规范
- 每个数据点必须标注来源
- 区分一手数据和二手数据
- 时效性标注（日期）

## 结论质量
- 每个结论 ≤ 3 个数据支撑
- 区分事实与推断
- 不确定性用概率表述（"高概率"/"待验证"）
```

### Phase 5: Decompose — 研究导向

```markdown
## Foundation
- [ ] 确定分析框架
- [ ] 收集公开数据/资料

## Core
- [ ] 逐维度分析（每个维度一个任务）
- [ ] 交叉验证关键发现
- [ ] 提炼核心洞察

## Polish
- [ ] 写执行摘要
- [ ] 可视化（图表）
- [ ] Review: 对照 Brief 检查
```

## 🎯 决策支持

### Phase 1: Clarify — 追问清单

```markdown
1. 要做的是什么决策？不可逆程度？
2. 有几个候选方案？各自的核心理由？
3. 决策的评判标准是什么？（成本？速度？风险？长期价值？）
4. 谁做最终决策？他/她的偏好倾向？
5. 截止时间？决策错误的后果？
6. 最坏情况是什么？能否承受？
7. 有没有不做决策的选项？
```

### Phase 2: Brief — 追加字段

```markdown
## 决策定义
- 决策问题：
- 决策者：
- 截止时间：

## 候选方案
| 方案 | 核心主张 | 支持者 |
|------|---------|--------|

## 评估标准（加权）
| 标准 | 权重 | 度量方法 |
|------|------|---------|
```

### Phase 3: Architect — 追加模块

```markdown
## 决策树
{各方案在不同条件下的结果}

## 评估矩阵
| 方案 | 标准A(0.4) | 标准B(0.3) | 标准C(0.3) | 总分 |
|------|-----------|-----------|-----------|------|
| 方案1 | | | | |
| 方案2 | | | | |
```

### Phase 4: Standards — 追加规范

```markdown
## 评估规范
- 每个标准用 1-5 打分
- 权重和为 1.0
- 标注主观判断 vs 客观数据
- 敏感度分析：改变权重是否改变结论？
```

### Phase 5: Decompose — 决策导向

```markdown
## Foundation
- [ ] 明确决策问题与选项
- [ ] 确定评估标准与权重

## Core
- [ ] 收集每个方案的数据
- [ ] 填写评估矩阵
- [ ] 敏感度分析

## Polish
- [ ] 写推荐结论
- [ ] 风险缓解计划
- [ ] Review: 对照 Brief 检查
```
